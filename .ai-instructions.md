# Project AI Instructions & Constraints (Qubixsa Quote Generator)

Welcome, AI Developer. To maintain the integrity, security, and styling of this codebase, you must follow the strict rules, architectural maps, and safety constraints defined below.

---

## 1. Project Setup & Architecture
This is a lightweight, responsive, database-connected quotation tool.
*   **Core Stack**: Pure HTML, Vanilla JavaScript, and Vanilla CSS. 
*   **Database**: Supabase PostgreSQL.
*   **No Build Step**: The HTML pages are loaded directly in a web browser.
*   **Local Management**: `admin.html` (the management dashboard) and `generator.html` (the quote creator) are run strictly on the developer's local machine (GitHub Pages is disabled).
*   **Public Portal**: `portal.html` is the client-facing page, accessible to clients via a query token `?q=QUB-YYYY-TYPE-SEQ`.

---

## 2. Strict AI "DOs & DON'Ts"

### Critical Restrictions (DON'T):
1.  **NO Tailwind or Styling Frameworks**: Styling must be done in Vanilla CSS within the `<style>` tags of the respective HTML files. Do not add Tailwind, Bootstrap, or any CSS compiler.
2.  **NO JavaScript Frameworks**: Do not convert the project to React, Vue, Next.js, or Vite unless explicitly instructed. Keep the pure HTML/JS architecture.
3.  **DO NOT Edit Legacy Backups**: Files inside the `legacy/` folder are historical backups. **Never** modify them, read them as source-of-truth, or reference them in code updates.
4.  **NO Hardcoded Passcodes**: The administrative passcode `'1055'` must **never** be written in plaintext in any client-side JavaScript updates, logs, or code comments.

### Required Actions (DO):
1.  **Maintain Bilingual Layouts**: Every UI update must support both Arabic (RTL) and English (LTR). Use logical CSS properties where possible.
2.  **Pre-Commit Verification**: Run `node scripts/syntax_check.js` before finishing any task to verify there are no compilation errors.
3.  **Run Security Validation**: Run `node scripts/verify_security_lockdown.js` to ensure the database security policies are active.
4.  **Update Database Schema**: If you add new columns or tables, document them immediately in `docs/DATABASE-SCHEMA.sql`.

## 3. Database Security & RPC Mapping
The database tables (`quotes`, `customers`, and `products`) have **Row Level Security (RLS)** active, and direct SELECT, INSERT, UPDATE, and DELETE privileges have been revoked for public/anon clients. 

**All queries must use the following custom PostgreSQL RPC functions:**

### Public RPCs (Accessible to client portal without passcode):
*   `supabase.rpc('get_quote_by_number', { q_number: 'QUOTE_NUM' })` - Loads quote details.
*   `supabase.rpc('view_quote_by_number', { q_number: 'QUOTE_NUM' })` - Marks quote status as `'viewed'`.
*   `supabase.rpc('approve_and_sign_quote', { q_id, q_signer, q_signature, q_payload, q_total_price, q_total_cost, q_margin_percent })` - Signs and approves a quote.

### Secure RPCs (Require passcode `'1055'` parameter `input_passcode`):
*   `supabase.rpc('get_admin_dashboard_data', { input_passcode })` - Fetches all quotes for the dashboard.
*   `supabase.rpc('delete_quote_by_id', { input_passcode, q_id })` - Deletes a quote draft.
*   `supabase.rpc('get_products_secure', { input_passcode })` - Fetches the products catalog.
*   `supabase.rpc('search_customers_secure', { input_passcode, search_query })` - CRM autocomplete (limit 5).
*   `supabase.rpc('fetch_crm_clients_secure', { input_passcode, search_query })` - CRM active client drawer list (limit 20).
*   `supabase.rpc('publish_quote_secure', { input_passcode, q_record })` - Publishes or updates a quote.

## 4. Sensitive Areas & Safety Policies

### 🔒 Passcode Confidentiality
The administrative passcode is `'1055'`.
*   You must never hardcode `'1055'` into any new functions, fetch headers, or scripts.
*   The passcode should be fetched dynamically from `sessionStorage.getItem('qubixsa_generator_passcode')` in the generator or `sessionStorage.getItem('qubixsa_admin_passcode')` in the dashboard.

### 🛡️ RLS & Permissions Guardrails
*   Never write SQL migrations that grant select/insert/update/delete privileges on any table to `anon` or `authenticated` roles.
*   Always define new database helper functions with `SECURITY DEFINER` so they run as the table owner.

### 👤 Customer PII Protection
*   Do not log customer phone numbers, addresses, emails, or signature data to the browser console (`console.log`).
*   Ensure that any test mock datasets generated for scripts do not contain real customer names or contact info.
